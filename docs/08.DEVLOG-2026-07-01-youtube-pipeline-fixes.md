## Bug Fixes

### 1. Celery ForkPoolWorker — `InterfaceError: asyncpg connection binding`
**Commit:** `bb8ffd8`  
**Severity:** Critical — seluruh pipeline komentar tidak pernah berjalan

**Akar masalah:**  
Celery menggunakan `ForkPoolWorker` (prefork). Ketika worker child process di-fork, ia mewarisi `AsyncEngine` + koneksi asyncpg yang terikat ke event loop proses parent. Saat child memanggil `asyncio.run()`, event loop baru dibuat — tapi koneksi asyncpg lama tidak valid di loop baru, menghasilkan:
```
InterfaceError: cannot perform operation: another operation is in progress
```

**Files yang diubah:**

| File | Perubahan |
|------|-----------|
| `app/services/collector/service.py` | Tambah parameter opsional `db: AsyncSession | None = None`. Jika `None`, buat session sendiri via `AsyncSessionLocal`. Jika diisi (dari Celery), gunakan langsung tanpa buat baru. |
| `app/workers/youtube_worker.py` | Tambah helper `_get_fresh_session()` — buat `create_async_engine()` baru per-task. Semua fungsi async (`_run_youtube_pipeline`, `_run_comments`, `_run_fetch_trending`) kini pakai engine baru dan pass `db=` ke `collect_for_platform()`. |
| `app/workers/collector_worker.py` | Tambah `_get_fresh_session()` yang sama. Refaktor `_run_collection()` untuk pass `db=` ke `CollectorService`. |

**Pola fix (`_get_fresh_session`):**
```python
def _get_fresh_session():
    from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine
    from app.shared.config import settings
    fresh_engine = create_async_engine(
        settings.database_url, pool_size=2, max_overflow=0, echo=False
    )
    session_factory = async_sessionmaker(
        bind=fresh_engine, class_=AsyncSession,
        expire_on_commit=False, autocommit=False, autoflush=False
    )
    return fresh_engine, session_factory
```

**Verifikasi:** Task `collect_youtube_pipeline_task` untuk keyword "piala dunia" berhasil dengan `comment_jobs_dispatched: 10` (sebelumnya selalu 0).

### 3. `INTERNAL_ERROR` pada `category_id` Invalid di `/videos/popular`
**Commit:** `5b07c85`  
**Severity:** High — endpoint crash dengan 500 saat Swagger default `"string"` atau typo kategori dikirim

**Akar masalah:**  
`YouTubeDataAPIClient.fetch_popular()` meneruskan `category_id` mentah ke Google API. Google menolak nilai non-numerik (contoh: `"string"`, `"oki"`) dengan HTTP 400. `httpx.HTTPStatusError` ini **tidak ditangkap di mana pun** sehingga jatuh ke handler generic FastAPI → `INTERNAL_ERROR`.

**Files yang diubah:**

| File | Perubahan |
|------|-----------|
| `app/integrations/youtube_data_api/client.py` | (1) Validasi `category_id.strip().isdigit()` — nilai non-digit diabaikan. (2) Tambah helper `_get()` yang membungkus semua `httpx` error jadi `ExternalAPIError`. (3) Terapkan `_get()` ke `search_videos()` dan `fetch_popular()`. (4) Tambah try/except `ExternalAPIError` di `list_comment_threads()`. |

**Pattern baru (konsisten dengan `EnsembleDataClient`):**
```python
async def _get(path: str, params: dict[str, Any]) -> httpx.Response:
    try:
        async with httpx.AsyncClient(timeout=30) as client:
            resp = await client.get(f"{_BASE_URL}/{path}", params=params)
            resp.raise_for_status()
            return resp
    except httpx.HTTPStatusError as exc:
        raise ExternalAPIError(
            service="YouTubeDataAPI",
            message=f"HTTP {exc.response.status_code}: {exc.response.text[:200]}",
        )
    except httpx.RequestError as exc:
        raise ExternalAPIError(service="YouTubeDataAPI", message=str(exc))
```

**Verifikasi:** `fetch_popular(category_id="string")` dan `category_id="oki"` → `OK, items=2` (tidak crash).

## Catatan Arsitektur

### Pola Error Handling yang Konsisten
Setelah sesi ini, semua HTTP client (`EnsembleDataClient` dan `YouTubeDataAPIClient`) membungkus error ke `ExternalAPIError(AppException)`. Error ini ditangkap oleh FastAPI exception handler di `app/main.py` dan dikembalikan sebagai response terstruktur (HTTP 502) — bukan `INTERNAL_ERROR` generik (HTTP 500).

### Pola Fresh Session untuk Celery
Setiap Celery task yang butuh DB **wajib** membuat engine baru via `_get_fresh_session()` dan pass session ke service layer lewat parameter `db=`. Ini adalah invariant yang harus dipertahankan untuk semua task baru.
